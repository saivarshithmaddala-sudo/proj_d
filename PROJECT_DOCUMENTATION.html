Zamazon Identity Engine

Technical Documentation & Architecture Specification

1. Executive Summary

In modern e-commerce and full-stack environments, users frequently interact with platforms using fragmented or partial contact details—such as utilizing alternative email addresses, ordering as guests, or signing up with different phone numbers. This fragmentation results in disparate contact entries belonging to the same physical person.

The Zamazon Identity Engine is a real-time, transaction-safe identity reconciliation engine. Given a stream of `/api/identify` requests carrying an email and/or phoneNumber, it automatically maps and consolidates these details into unified, exactly **two-level deep identity clusters**: a single root primary contact, and multiple linked secondary contacts.

2. Architecture & Tech Stack

  • Framework: Next.js 14 (App Router) for Serverless API route handling.
  • Database: MongoDB Atlas (Cloud replica set with transaction support).
  • ORM: Prisma v6 for type-safe database queries.
  • Language: TypeScript (strict mode) for compile-time interface safety.
  • Testing: Vitest for integration and edge-case execution.

3. Schema Design

The system maps contacts dynamically using a self-referencing relationship inside a single MongoDB collection. Each secondary contact holds a reference to its primary parent via linkedId.

// prisma/schema.prisma

enum LinkPrecedence {
  primary
  secondary
}

model Contact {
  id             String         @id @default(auto()) @map("_id") @db.ObjectId
  phoneNumber    String?
  email          String?
  linkedId       String?        @db.ObjectId // Reference to the primary contact
  linkedContact  Contact?       @relation("ContactLinks", fields: [linkedId], references: [id], onDelete: NoAction, onUpdate: NoAction)
  secondaryLinks Contact[]      @relation("ContactLinks")
  linkPrecedence LinkPrecedence @default(primary)
  createdAt      DateTime       @default(now())
  updatedAt      DateTime       @updatedAt
  deletedAt      DateTime?

  @@index([email])
  @@index([phoneNumber])
  @@index([linkedId])
}

4. Core Reconciliation Algorithm

When an /api/identify request is triggered, the engine resolves the identity state via the following workflow:

  1. Match Detection: Queries the database for existing records containing the given email or phone number.
  2. Root Pathing: Recursively walks the parent chain for all matching records to find the single root primary parent ID.
  3. Cluster Merging: If matching records span multiple distinct primary clusters, the oldest primary contact survives. All younger primary contacts are updated to secondary precedence and link directly to the oldest primary. Any children previously pointing to the younger primaries are cascade-repointed to the oldest primary.
  4. New Information Checks: If the request introduces a new email or phone number not yet represented in the resolved cluster, a new secondary contact is created.
  5. Response Formatting: The engine structures the output, listing the primary contact's details first, followed by secondary details in ascending chronological order.

5. Concurrency Control

To avoid race conditions during concurrent requests (e.g., when two identical requests for a new customer arrive at the exact same millisecond), the engine runs queries inside a database transaction.

Retry-On-Conflict Strategy

If MongoDB throws a transient WriteConflict error (which occurs when two concurrent transactions attempt to write to the same document range), the application catches the error, waits a random delay to prevent locks, and retries. On the second attempt, the match lookup succeeds, avoiding the creation of duplicate primary parents.

6. Test Coverage

The integration test suite executes 10 unique scenarios validating functionality:

  • New Identity: Creates primary contact on first-ever request.
  • Information Extension: Sharing only phone or only email creates a secondary contact.
  • Cluster Merge: Combines separate identities under the oldest parent.
  • Exact Duplicate: Short-circuits and ignores redundant inputs.
  • Type Coercion: Converts numeric phone inputs into strings dynamically.
  • Race Condition Check: Validates concurrent requests execution safety.
